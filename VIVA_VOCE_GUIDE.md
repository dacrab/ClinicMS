## Ερωτήσεις που μπορεί να γίνουν

### 1. OOP Concepts

**Q: Ποιες OOP αρχές εφαρμόσατε;**
A: 
- **Encapsulation**: Όλα τα πεδία είναι private με getters/setters
- **Inheritance**: Η κλάση `Specialist` επεκτείνει την `Doctor`
- **Polymorphism**: Το `DataStore` αποθηκεύει `List<Doctor>` που περιέχει και `Specialist` objects
- **Singleton**: Το `DataStore` χρησιμοποιεί το Singleton pattern
- **Enum**: Το `Appointment.Status` για type-safe καταστάσεις

**Q: Γιατί χρησιμοποιήσατε Singleton για το DataStore;**
A: Για να εξασφαλίσω ότι υπάρχει μόνο μία παρουσία του DataStore σε όλη την εφαρμογή, ώστε όλοι οι controllers να μοιράζονται τα ίδια δεδομένα.

**Q: Πώς λειτουργεί η κληρονομικότητα στο Specialist;**
A: Το `Specialist` επεκτείνει το `Doctor` και προσθέτει δύο επιπλέον πεδία: `subSpecialty` και `maxWeeklyPatients`. Κληρονομεί όλα τα πεδία και μεθόδους του `Doctor`.

### 2. File I/O

**Q: Πώς αποθηκεύετε τα δεδομένα;**
A: Χρησιμοποιώ CSV αρχεία στον φάκελο `data/`. Κάθε φορά που γίνεται προσθήκη, ενημέρωση ή διαγραφή, το `DataStore` επανεγγράφει ολόκληρη τη συλλογή στο αντίστοιχο CSV αρχείο.

**Q: Πώς χειρίζεστε τα κόμματα μέσα στα πεδία;**
A: Χρησιμοποιώ escape character `&#44;` για να αντικαταστήσω τα κόμματα πριν την αποθήκευση και unescape κατά την ανάγνωση.

**Q: Τι γίνεται αν το αρχείο δεν υπάρχει;**
A: Το `DataStore` δημιουργεί αυτόματα τον φάκελο `data/` και τα αρχεία CSV αν δεν υπάρχουν.

### 3. Validation

**Q: Ποιους ελέγχους κάνετε στα δεδομένα;**
A: 
- Ημερομηνίες σε μορφή `ηη/ΜΜ/εεεε`
- Ώρες σε μορφή `ΩΩ:ΛΛ`
- Email με regex pattern
- Τηλέφωνο με regex pattern
- Υποχρεωτικά πεδία (όνομα, ειδικότητα κλπ.)

**Q: Πού γίνονται οι έλεγχοι;**
A: Στην κλάση `Validator` που περιέχει στατικές μεθόδους για κάθε τύπο ελέγχου.

### 4. Double Booking Prevention

**Q: Πώς αποτρέπετε τη διπλοκράτηση;**
A: Στο `DataStore`, η μέθοδος `checkDoubleBooking()` ελέγχει αν ο ιατρός έχει ήδη ραντεβού την ίδια ημερομηνία και ώρα. Αν ναι, ρίχνει `IllegalStateException`.

**Q: Τι γίνεται με τα ακυρωμένα ραντεβού;**
A: Τα ακυρωμένα ραντεβού δεν λαμβάνονται υπόψη στον έλεγχο διπλοκράτησης.

### 5. JavaFX

**Q: Πώς οργανώσατε το UI;**
A: Χρησιμοποιώ FXML αρχεία για τη δήλωση της διεπαφής και Controllers για τη λογική. Κάθε view έχει το δικό του FXML και Controller.

**Q: Πώς εφαρμόσατε το styling;**
A: Με ένα κεντρικό CSS αρχείο (`style.css`) που περιέχει όλα τα styles για την εφαρμογή.

**Q: Πώς λειτουργεί η πλοήγηση;**
A: Ο `MainController` φορτώνει δυναμικά τα διαφορετικά views στο `contentArea` με βάση το κουμπί που πατήθηκε στην πλαϊνή μπάρα.

### 6. Error Handling

**Q: Πώς χειρίζεστε τα σφάλματα;**
A: Χρησιμοποιώ try-catch blocks σε όλες τις κρίσιμες λειτουργίες και εμφανίζω Alert dialogs με κατάλληλα μηνύματα στον χρήστη.

**Q: Τι γίνεται αν ο χρήστης προσπαθήσει να διαγράψει ιατρό με ενεργά ραντεβού;**
A: Το σύστημα ελέγχει αν υπάρχουν προγραμματισμένα ραντεβού και εμφανίζει μήνυμα σφάλματος χωρίς να επιτρέψει τη διαγραφή.

### 7. ID Generation

**Q: Πώς δημιουργείτε τα IDs;**
A: Χρησιμοποιώ την κλάση `IdGenerator` που χρησιμοποιεί `AtomicInteger` για thread-safe αυτόματη αρίθμηση. Κατά τη φόρτωση δεδομένων, ο μετρητής αρχικοποιείται στο μέγιστο υπάρχον ID.

### 2. Double Booking Check
```java
private void checkDoubleBooking(Appointment incoming, int excludeId) {
    if (incoming.getStatus() == Appointment.Status.CANCELLED) return;
    boolean conflict = appointments.stream()
            .filter(a -> a.getId() != excludeId)
            .anyMatch(a -> a.conflictsWith(incoming));
    if (conflict) {
        throw new IllegalStateException("...");
    }
}
```

### 3. CSV Serialization
```java
public String toCsv() {
    return id + "," + escape(name) + "," + escape(specialty) + "," + 
           escape(phone) + "," + escape(email);
}

private static String escape(String s) {
    return s == null ? "" : s.replace(",", "&#44;");
}
```

### 4. Validation
```java
public static boolean isValidDate(String date) {
    return date != null && DATE_PATTERN.matcher(date).matches();
}
```
